# Scan Report

## The Problem

`SpecularProvider.g.cs` tells the runtime *what* to return from `GetAssemblies`/`GetTypes`/`Scan`, but it doesn't give you anything to introspect at development time: there's no structured way to answer "which selectors did I write, what did each one resolve to, and where did those types come from?" without reading generated switch-statement bodies by hand.

## The Solution: `SpecularScanReport.g.cs`

Alongside `SpecularProvider.g.cs`, the generator emits a second, independent file — `SpecularScanReport.g.cs` — containing an `internal static class SpecularScanReport` with:

- **`Entries`** — an `IReadOnlyList<ScannerExpression>`, one entry per `GetAssemblies`/`GetTypes`/`Scan` call site in your code, each carrying:
  - `Kind` — `Assembly`, `Reflection`, or `ServiceDescriptor`, matching which method was called.
  - `FilePath`/`LineNumber` — the call site's source location.
  - `SelectorText` — the selector lambda's literal source text (e.g. `z => z.FromAssemblyOf<IService>().AddClasses()...`).
  - `DiscoveredAssemblies`/`DiscoveredTypes` — what the selector actually resolved to, as real `typeof()`/assembly references where possible.
- **`MermaidDiagram`** — a `const string` rendering the same entries as a `flowchart TD`, built from identical data.

Both are generated purely from data the generator already resolves while building `SpecularProvider.g.cs` — the scan report doesn't perform any additional symbol resolution of its own.

## Example

Given:

```csharp
provider.GetTypes(z => z.FromAssemblyOf<IService>().GetTypes(x => x.AssignableTo<IService>()));
```

the generator emits an entry like:

```csharp
new(
    ScannerExpressionKind.Reflection,
    "Program.cs",
    13,
    "z => z.FromAssemblyOf<IService>().GetTypes(x => x.AssignableTo<IService>())",
    new ScanReportAssembly[] { },
    new ScanReportType[]
    {
        new("MyAssembly", "MyAssembly.IService", typeof(MyAssembly.IService)),
        new("MyAssembly", "MyAssembly.Service", typeof(MyAssembly.Service)),
        new("MyAssembly", "MyAssembly.InternalService", null), // non-public: name only, see below
    }
)
```

and `SpecularScanReport.MermaidDiagram` includes a corresponding `flowchart TD` fragment linking the call site to the expression to each discovered type.

## The `Type: null` Fallback

`ScanReportType.Type` and `ScanReportAssembly.Assembly` are only populated with a real `typeof()`/assembly reference when the discovered type is accessible from your compilation (the same accessibility check the main provider uses for its public-type fast path). For non-public types — like `InternalService` above — `Type` is `null` and you fall back to the always-present `AssemblyName`/`TypeName` strings.

This is intentional, not a limitation to work around: the scan report never uses the private-assembly reflection machinery `SpecularProvider.g.cs` uses for non-public types, so it introduces **no new trim/AOT surface**. Every `typeof()` it emits is one the main provider would already prove reachable.

## Using It in Tests

Because `SpecularScanReport` is a normal (if `internal`) generated type, it's usable directly from unit or snapshot tests in the same project — no special test infrastructure required:

```csharp
SpecularScanReport.Entries.ShouldContain(e => e.DiscoveredTypes.Any(t => t.Type == typeof(MyService)));
```

## Opting Out

The scan report is generated by default. To disable it — for example, in a library that doesn't want the extra generated file — set:

```xml
<PropertyGroup>
  <SpecularGenerateDiagnostics>false</SpecularGenerateDiagnostics>
</PropertyGroup>
```

## Known Limitation: Warm Caches After Upgrading

`SelectorText`, `DiscoveredTypes`, and `DiscoveredAssemblies` are new fields added to the same records that back the [cross-assembly cache](./cross-assembly-caching.md) (`SpecularProvider.ctpjson`). A cache file written by a pre-upgrade version of the generator won't have these fields populated. If you upgrade Specular on a solution with a warm `obj/ctp` cache, entries resolved from that stale cache may show an empty `SelectorText` or `DiscoveredTypes` until that selector's cache entry is invalidated (its source text changes) or you do a clean rebuild.
