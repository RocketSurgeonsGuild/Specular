# Phase 1 Data Model: Sample Applications, AOT Validation & Documentation Build Pipeline

**Feature**: 004-sample-applications | **Date**: 2026-06-27

This feature is infrastructure/sample-heavy, not a data-modeled service. The "entities" below are
**inventory and configuration models** — the durable, structured artifacts the implementation
produces and the pipeline reasons about. They map directly to the spec's Key Entities (FR-001…029,
SC-001…011). No relational schema, persistence, or migrations are implied.

A companion `data-model-diagram.mmd` (Mermaid) is generated by the `data-model-diagram` hook.

---

## Entity: SampleLibrary

A reusable, general-purpose library under `samples/libraries/` whose services Indago discovers.

| Field               | Type            | Rules / Notes                                         |
| ------------------- | --------------- | ----------------------------------------------------- |
| `name`              | string          | e.g. `Indago.Samples.Catalog`. Unique.                |
| `targetFrameworks`  | string          | `net8.0;net10.0` (repo multi-target).                 |
| `registrationStyle` | enum            | `interface-matching` \| `attribute-based` \| `mixed`. |
| `services`          | SampleService[] | **3–6** entries (FR-002).                             |
| `isPackable`        | bool            | Always `false` (excluded from shipped surface).       |
| `changesPublicApi`  | bool            | Always `false` (Principle III; RS0017 stays green).   |

**Validation**: exactly **three** libraries (FR-001). Collectively they MUST cover
interface-matching **and** attribute-based registration **and** ≥2 distinct lifetimes **and** ≥1
opt-out type (FR-003/004, SC-001).

### Inventory (planned)

| Library                        | Style                                                                        | Lifetimes shown       | Opt-out type?                   |
| ------------------------------ | ---------------------------------------------------------------------------- | --------------------- | ------------------------------- |
| `Indago.Samples.Catalog`       | interface-matching (`FromAssemblyOf<>().AddClasses().AsMatchingInterface()`) | Singleton (default)   | no                              |
| `Indago.Samples.Notifications` | attribute-based (`[ServiceRegistration(...)]`)                               | Scoped + Transient    | no                              |
| `Indago.Samples.Diagnostics`   | mixed                                                                        | Singleton + Transient | **yes** (`[ExcludeFromIndago]`) |

## Entity: SampleService

A service/implementation pair within a `SampleLibrary` that Indago is expected to discover.

| Field                  | Type | Rules / Notes                                            |
| ---------------------- | ---- | -------------------------------------------------------- |
| `serviceType`          | type | The contract/interface (or self) registered.             |
| `implementationType`   | type | Concrete implementation.                                 |
| `lifetime`             | enum | `Singleton` \| `Scoped` \| `Transient`.                  |
| `registrationStyle`    | enum | `interface-matching` \| `attribute-based`.               |
| `excludedFromScanning` | bool | `true` only for the opt-out demonstration type (FR-004). |

**Validation**: a service with `excludedFromScanning=true` MUST NOT appear in any host's
discovered set, across every host and publish mode (Edge Case "Excluded/opt-out types").

## Entity: HostApplication

A runnable demonstration consuming the sample libraries and performing Indago scanning.

| Field                 | Type         | Rules / Notes                                                             |
| --------------------- | ------------ | ------------------------------------------------------------------------- |
| `name`                | string       | Unique.                                                                   |
| `kind`                | enum         | `console` \| `web-minimal-api` \| `blazor-wasm` \| `maui`.                |
| `priority`            | enum         | `P1` \| `P2` \| `P3`.                                                     |
| `aotMode`             | enum         | `native-aot` \| `wasm-aot` \| `platform-maui`.                            |
| `aotPublishTarget`    | string       | `net10.0` × current-OS RID (MAUI: `net10.0-android`).                     |
| `expectedServiceSet`  | ServiceRef[] | Hard-coded expected discovery result (FR-008).                            |
| `assertionPoint`      | enum         | `startup` (console/web) \| `on-load/launch` (blazor/maui).                |
| `failureSignal`       | enum         | `nonzero-exit` (console/web) \| `detectable-failure-state` (blazor/maui). |
| `isDemonstrationHost` | bool         | `true` for the four hosts; `false` for the negative fixture.              |

**Validation**: exactly **four** demonstration hosts (FR-006). Each consumes the shared libraries
with no per-host library modification (FR-005, Edge Case "New host, no extra wiring"). Each fails
visibly on service-set mismatch (FR-010, SC-002). MAUI CI target is `net10.0-android` only.

## Entity: AotPublishValidation

A pipeline-executed verification that a demonstration host AOT/trim-publishes cleanly.

| Field           | Type            | Rules / Notes                                                                                         |
| --------------- | --------------- | ----------------------------------------------------------------------------------------------------- |
| `host`          | HostApplication | The host being published.                                                                             |
| `target`        | string          | `net10.0` × current-OS RID (single deterministic matrix, FR-013).                                     |
| `warningPolicy` | enum            | `warnings-as-errors` (zero-warning, FR-012).                                                          |
| `outcome`       | enum            | `pass` \| `fail` \| `skip` (FR-016/017).                                                              |
| `skipReason`    | string?         | Required when `outcome=skip` (e.g. missing `wasm-tools`/MAUI workload, or non-android MAUI platform). |
| `isNamedStep`   | bool            | Always `true` — surfaced as a discrete named pipeline step (FR-014, SC-006).                          |

**State transitions** (per host/target):

```
            ┌─ toolchain present ─► run publish ─► clean    ► PASS
 evaluate ──┤                                   └─ any warn ► FAIL
            └─ toolchain absent  ─────────────────────────► SKIP (with reason)
```

**Validation**: `skip` is never reported as `pass` (FR-016, SC-005). Non-android MAUI platforms
(iOS/MacCatalyst/Windows) are always `skip` with an out-of-CI-scope reason (FR-006/016, SC-005).

## Entity: NegativeFixtureHost

A permanent fixture that intentionally emits an Indago-attributable trim/AOT warning.

| Field                   | Type      | Rules / Notes                                                                                                                   |
| ----------------------- | --------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `name`                  | string    | `Indago.Samples.NegativeFixture`.                                                                                               |
| `excludedFromNormalAot` | bool      | Always `true` (FR-020).                                                                                                         |
| `isDemonstrationHost`   | bool      | Always `false` (not one of the four).                                                                                           |
| `expectedWarning`       | signature | The Indago-attributable trim/AOT warning the inverted assertion matches (signature pinned during implementation — research D2). |
| `assertion`             | enum      | `inverted` — step passes only when the publish warns/fails (FR-021).                                                            |
| `exemptFromZeroWarning` | bool      | Always `true` — the sole exemption (FR-012/021, SC-003).                                                                        |

**State transitions**:

```
 publish fixture (net10.0 × current RID) ─┬─ expected warning present ► PASS
                                          └─ publish clean / warning gone ► FAIL (regression detector broken)
```

## Entity: DocsBuildModule

A named ModularPipelines module in `build/Build.cs` producing the docs artifact.

| Field               | Type     | Rules / Notes                                                                      |
| ------------------- | -------- | ---------------------------------------------------------------------------------- |
| `name`              | string   | e.g. `DocsModule`.                                                                 |
| `dependsOn`         | Module[] | `BuildSolution` (Release bin must exist).                                          |
| `apiSource`         | path     | `src/Indago/bin/Release/net8.0/Indago.dll` + `Indago.xml` (FR-023; NOT a publish). |
| `apiGenerator`      | tool     | `xmldocmd@2.9.0` (mise dotnet tool).                                               |
| `frontmatterStep`   | script   | `docs/scripts/add-api-frontmatter.mjs` (Node).                                     |
| `siteBuilder`       | tool     | `npm ci` + `npm run build` (Astro/Starlight) via `context.Node()`.                 |
| `outputDir`         | folder   | `ArtifactSettings.ArtifactsDirectory / "docs"` (FR-024).                           |
| `deploys`           | bool     | Always `false` — deploy-agnostic (FR-028).                                         |
| `failsIfXmlMissing` | bool     | Always `true` (Edge Case "Docs build consumes bin").                               |

**Validation**: no separate `dotnet publish` for docs (FR-023, SC-008); built site lands in
`artifacts/docs` (FR-024); reuses extensions building blocks (FR-025).

## Entity: CiDocsDeploy (GitHub Actions, not the pipeline)

| Field              | Type     | Rules / Notes                                                     |
| ------------------ | -------- | ----------------------------------------------------------------- |
| `artifactPath`     | path     | `artifacts/docs`.                                                 |
| `actions`          | string[] | `actions/upload-pages-artifact`, `actions/deploy-pages` (FR-028). |
| `gatedToBranch`    | string   | `main` only (FR-027, SC-010).                                     |
| `runsOnEveryBuild` | bool     | Docs **built** every CI run; **deployed** only from `main`.       |

**Validation**: deploy from non-`main` is a defect; failing to deploy from `main` is a defect
(Edge Case "Pages deploy gating").

## Relationships (summary)

- `SampleLibrary 1—* SampleService`
- `HostApplication *—* SampleLibrary` (each host consumes all three libraries)
- `HostApplication 1—1 AotPublishValidation` (per AOT-capable demonstration host)
- `NegativeFixtureHost 1—1 (inverted) AotPublishValidation`
- `DocsBuildModule 1—1 CiDocsDeploy` (artifact handed to CI; deploy gated to `main`)
- `DocsBuildModule —depends-on→ BuildSolution`
